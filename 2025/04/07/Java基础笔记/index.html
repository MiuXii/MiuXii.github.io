JVM

解释、编译和运行

解释型：程序执行时，逐行解释源代码，不生成独立可执行的文件，通常由解释器动态执行，跨平台性好，但执行速度相对慢（python、JS）
编译型：程序执行前，整个源代码会被编译成机器码或字节码，生成可执行文件，执行时直接运行编译后的代码（C、C++）

  • 编译：Java源代码文件（.java）通过编译器被编译成字节码文件（.class）。在这个过程中，编译器会检查语法错误、语义错误等，将源代码转换为JVM能够理解的字节码指令。
  • 运行Java程序时，JVM会启动，然后进行类加载等一系列操作。JVM会将字节码文件加载到内存中，然后通过解释器（执行字节码指令，从而实现程序的运行。

JDK和JRE

JDK>JRE>JVM

  • JDK‌：Java开发工具包，包括编译工具（javac.exe）、打包工具（jar.exe）等，也包含JRE。JDK是开发Java程序的主要工具包，包括了Java运行环境、Java工具和Java基础的类库（Java API）。
  • ‌JRE‌：Java运行环境，包括JVM标准实现（JVM）和Java核心类库。JRE是运行Java程序必不可少的运行时环境。

内存模型

  • 线程共享（大家都可以知道）：方法区和堆
  • 线程私有（分家了，自己的事情）：虚拟机栈，本地方法栈和程序计数器

堆：存放了对象实例及数组。
方法区(JDK8以后叫元空间,使用本地内存)：存储已被虚拟机加载的类型信息、常量、静态变量、即时编译器编译后的代码缓存等数据。
虚拟机栈：每个线程在创建时都会创建一个虚拟机栈，其内部保存一个个的栈帧（Stack Frame），对应着一次次的Java方法调用。每个方法被执行的时候都会创建一个”栈帧”,用于存储局部变量表(包括参数)、操作栈、方法出口等信息。

全局变量在堆，局部变量在栈

本地方法栈：与虚拟机栈所发挥的作用是非常相似的，其区别不过是虚拟机栈为虚拟机执行Java 方法（也就是字节码）服务，而本地方法栈则是为虚拟机使用到的Native 方法服务。
程序计数器：记录着线程当前正在执行的指令的位置。

垃圾回收

自动识别并回收不再使用的对象，释放内存空间。

  • 新生代——复制算法(当一块内存满了时，将存活的对象复制到另一块内存中，然后清空当前内存块)

    新生代的对象生命周期短，垃圾回收频繁，因此使用复制算法可以高效回收对象

    • 对象生命周期较短，大多数对象在Minor GC时会被回收。当Eden区满了时，触发Minor GC，将存活的对象复制到Survivor区，然后清空Eden区。如果一个对象在Survivor区中经过多次GC仍然存活，会被晋升到老年代。
    • 两个survivor区的作用：两个Survivor区交替使用，一个用于存放当前存活的对象，另一个用于下一次GC时存放存活的对象。每次对象从一个Survivor区复制到另一个Survivor区时，对象的“年龄”会增加。

  • 老年代——标记压缩、清除算法

    老年代的对象生命周期长，垃圾回收频率低，因此使用标记-压缩算法可以有效管理大内存空间并解决内存碎片问题

    • 标记清除：首先标记出所需回收的对象，在标记完成后统一回收掉所有被标记的对象。
    • 标记压缩：先标记存活对象，然后不是直接对可回收对象进行清理，而是让所有的对象都向一端移动，然后直接清理掉端边界以外的内存。

类加载器

当程序主动使用某个类时，如果该类还未被加载到内存中，则JVM会通过加载、连接、初始化3个步骤来对该类进行初始化(将字节码文件加载到内存中，并将其转换为java.lang.Class的实例)

  • 加载：类的全限定名获取类的字节码流，将字节码流转换为方法区中的运行时类信息。
  • 连接：检查准确性、分配内存等
  • 初始化：执行静态方法、构造方法等

双亲委派机制

当一个类加载器收到类加载请求时，它会先将该请求委派给它的父类加载器去尝试加载。只有当父类加载器无法加载该类时，子类加载器才会尝试加载。(核心库里有的直接加载，没有的再自己加载)
这种机制的设计目的是为了保证类的加载是有序的，避免重复加载同一个类。
双亲委派机制的工作过程：

  • 检查缓存：类加载器首先检查是否已经加载过该类，如果已经加载过，则直接返回该类。
  • 委派父加载器：如果没有加载过该类，类加载器将加载请求委派给父类加载器。
  • 递归过程：父类加载器也按照相同的过程处理加载请求，直到委派到启动类加载器（Bootstrap ClassLoader）。
  • 启动类加载器加载：启动类加载器尝试加载类，如果加载成功，则返回该类，否则继续步骤5。
  • 逐级回退：如果启动类加载器无法加载该类，加载请求逐级回退到下一级类加载器，直至回退到最初的请求者。
  • 自行加载：最终，如果所有父类加载器都无法加载该类，子类加载器才会尝试自行加载。

创建对象发生了什么

  • 0、类没在JVM内存里，类加载器加载
  • 1、（初始化）静态变量和静态代码块按顺序执行，且一定先于main发方法：在类加载时，静态变量和静态代码块按它们在类中出现的顺序依次执行。静态变量和静态代码块只在类第一次加载时执行一次。
  • 2、实例变量和实例代码块按顺序执行：在创建对象时，实例变量和实例代码块按它们在类中出现的顺序依次执行。每次创建对象时，实例变量和实例代码块都会执行一次。
  • 3、构造方法执行：实例变量和实例代码块执行完之后，才会执行构造方法。构造方法是在每次创建对象时执行的。

java基础

关键字

public

  • public修饰的方法可以被所有其他类所访问，它是最宽松的访问权限。它往往用于对外的情况，也就是对象或类对外的一种接口的形式
  • 在一个.java源文件中，直接定义的类只能有一个public，并且该public类必须与源文件同名，区分大小写。

private

private修饰的方法只能被自己访问和修改，它是最严格的访问权限。它是一种封装的体现，例如，大多数成员变量都是用private修饰的，它们不希望被其他任何外部的类访问如果想要访问被private封装的属性，则需要使用getter和setter两种方法。

final

  • 修饰类，此类不能被继承(String、StringBuffer、StringBuilder等都不可被继承；工具类也不可被继承，所以封装工具类可以用final关键字修饰)
  • 修饰方法不能被重写，子类继承父类final修饰的方法，重写会报错
  • 修饰成员变量，此变量只能被赋值一次
  • 修饰基本类型的

abstract

  • 抽象类提供了继承的概念，它的出发点就是为了继承。
  • 抽象类不能被标记为 final 或 static，因为它需要被其他类继承和实现。
  • 抽象类不能被实例化，实例化的工作应该交由继承它的子类来完成；抽象类中可以包含抽象方法但不能有方法体和大括号，当然也可以不包含抽象方法，只要包含一个抽象方法的抽象类，该方法必须要定义成抽象类。

interface

  • 抽象类的延伸，Java 8以前接口只有定义，不能有方法的实现；java 1.8中可以定义default方法体，这样避免添加方法后所有实现类都要修改的问题。
  • 接口中定义的方法默认就是公共的（public）。这意味着这些方法在实现类中应该是可以被公开调用的。
  • 接口中字段要static和final修饰。静态字段用于直接访问，表示接口本身的特性，而final字段作为常量确保接口中定义的不变性，防止实现类修改接口规范。

volatile

用于修饰变量，表示该变量可能会被多个线程同时访问并修改。

  • 可见性
    在多线程环境中，每个线程都有自己的工作内存（CPU 缓存），变量会从主内存拷贝到线程的工作内存中。普通变量的修改可能不会立即写回主内存，其他线程也就无法立即看到最新值；而 volatile 修饰的变量，每次读取都直接从主内存中读取，每次写入都立即写回主内存，从而保证了可见性。
  • 有序性
    多线程下执行同一代码可能会乱序指令进行性能优化。
    volatile 变量的读写操作，会在底层插入内存屏障（Memory Barrier），从而禁止指令重排序。
    • 写屏障（Store Barrier）：保证写操作之前的所有指令都执行完毕，才能执行写操作。
    • 读屏障（Load Barrier）：保证读操作之后的所有指令都必须在读操作之后执行。
  • 无法保证原子性

    原子性：一个操作要么全部执行成功，要么全部不执行，不会被其他线程中断或干扰。

    实际上，count++ 在底层分为三步：
    读取当前 count 的值（从主内存）
    将值加 1
    将新值写回 count（写回主内存）

假设线程 A 和线程 B 同时执行 count++。
两个线程可能同时读取到 count = 0。
然后各自加 1，得到 count = 1。
最后都写回主内存，结果是 count = 1，而不是期望的 2。

synchronized

用于保证共享资源的互斥访问，即同一时间只有一个线程能够访问被synchronized修饰的代码块或方法。

  • 可见性
  • 原子性
  • 性能开销大

面向对象特性

重写

子类重写父类方法，那么子类中该方法的访问级别不得低于父类访问级别。
比如：Object类有个toString()方法，开始重写这个方法的时候我们总容易忘记public修饰符，编译器当然不会放过任何教训我们的机会。出错的原因就是：没有加任何访问修饰符的方法具有包访问权限(protected)，包访问权限比public当然要严格了，所以编译器会报错的。

重载

存在于同一个类中，一个方法与已经存在的方法名称一致，但参数类型、个数、顺序至少有一个不同。

基本数据类型

整数类型

Java的整数类型包括：byte(1字节)、short(2字节)、int(4字节)、long(8字节)

隐式类型转换

Java 中的隐式类型转换遵循以下规则（按优先级从低到高）：
byte → short → int → long → float → double
char → int → long → float → double

如果不显式转型或有默认转型的机制，则报错

  • byte 1个字节，int4个字节
  • float a = 1.1报错
    1.1字面量属于double，非显式条件下(前面加(float))不能向下转型，float a =1.1f才正确
  • short b = 1 报错
    1字面量属于int,同上，short = (short)1才正确

+=,-=,*=,/=等复合运算符默认会自动进行强制类型转换，以+=为例，x += y等价于x = (T)(x + y)保证始终和 x 一致，避免类型不匹配的问题。

StringBuffer和StringBuilder

String不可变，线程安全,但字符串拼接会有性能问题，每次拼接都会产生一个新的对象
StringBuffer 适合多线程下，线程安全的，但是效率低些，因为加了 synchronized 关键字
StringBuilder适合单线程，线程不安全，但是速度快

  • 扩容机制
    当创建一个新的StringBuilder/StringBuffer对象时，如果没有指定容量，它通常会有一个默认容量，通常是16个字符。扩容过程：当StringBuilder中的字符数达到当前容量时，就需要进行扩容。扩容的过程通常是将容量增加到当前容量的2倍加2，即新容量 = 当前容量 * 2 + 2。

参数传递机制——值传递

值传递：方法调用时，实际参数的值会被复制一份传递给方法的形参。方法内部对形参的修改，不会影响原始的实际参数。
对于基本数据类型的引用都是值传递（值的复制），对于引用类型的引用也是值传递，这里传递的是对象引用的副本（即地址值的副本）。

  • 方法内部可以通过这个副本访问和修改对象的属性。
  • 但如果将形参重新指向一个新的对象，不会影响原始引用。

异常处理

  1. finally块无论如何都会执行
  2. 即使catch块中有return语句,finally块仍会执行
  3. catch块中的return会阻止finally块之后的代码执行

Object通用方法

==、equals()、hashCode()

  • 基本数据类型：基本数据类型只能使用 == 来比较是否相等，因为基本数据类型是没有 equals 方法的，比较的是两个基本数据类型变量的值是否相等。

  • 引用数据类型：当使用 == 的时候是比较两个变量的内存地址是否相等;

  • equals 是 Object 类中的方法，如果我们不重写 equals 方法那本质上还是使用 == 比较内存地址：

  • String 类中重写了equals(),比较的是变量的值（内容），和内存地址没有关系(地址相同，直接为true;不同，再判断内容)

  • hashCode方法的作用是获取哈希码，也称为散列码。定义在Object类中，默认是将对象的内存地址转换为整数之后返回。哈希码主要在HashMap、HashSet这类集合映射的时候用到。

  • Java 中的哈希集合（如 HashMap、HashSet）依赖于 hashCode() 来确定对象在哈希表中的位置，然后使用 equals() 来比较对象是否相等。

    • 如果只重写了 equals() 而没有重写 hashCode()，那么相等的对象可能会有不同的哈希值，导致哈希集合无法正确识别它们为同一个对象。
    • 如果只重写了 hashCode() 而没有重写 equals()，那么哈希值相同的对象在比较时可能被认为不相等，导致逻辑错误。

clone()

对象创建：1.new;
2.clone

  • 浅拷贝：创建一个新的对象，但是只复制原始对象的基本数据类型的字段或引用（地址），而不复制引用指向的对象。这意味着新对象和原始对象中的引用指向相同的对象。
    • 如果原始对象中的字段是基本数据类型，那么这些字段会被复制到新对象中，而如果字段是引用类型，则新对象和原始对- 象的对应字段将引用同一个对象。因此，对新对象所做的修改可能会影响到原始对象，因为它们共享相同的引用。
  • 深拷贝：创建一个新的对象，并且递归地复制原始对象的所有字段和引用指向的对象，而不仅仅是复制引用本身。
    • 深拷贝会递归复制整个对象结构，包括对象内部的对象，确保新对象和原始对象之间的所有关系都是独立的。这意味着对新对象所做的修改不会影响到原始对象，因为它们拥有彼此独立的副本。
